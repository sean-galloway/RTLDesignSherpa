# DFI v2.1 Master Interface

The controller drives DFI v2.1 master signals to the PHY. DFI v2.1 is the first DFI revision that supports LPDDR2; this controller targets that revision.

## Per-Phase Signal Naming

DFI v2.1 §3.1 defines per-phase signals with `_pN` suffix where N is the phase index. For `N_PHASES = 4`, signals exist as `_p0`, `_p1`, `_p2`, `_p3`. For `N_PHASES = 1`, signals can elide the suffix; this controller always emits the suffix for clarity.

## Command Interface (MC → PHY)

| Signal         | Width        | Notes                                                |
|----------------|--------------|------------------------------------------------------|
| `dfi_address`  | `DFI_ADDR_WIDTH` | DDR2: row/col address; LPDDR2: 20-bit packed CA word |
| `dfi_bank`     | `BANK_WIDTH` | DDR2: bank; LPDDR2: held idle (per Table 1)          |
| `dfi_cas_n`    | 1            | DDR2: CAS encoding; LPDDR2: held idle                |
| `dfi_cke`      | 1            | Clock enable                                         |
| `dfi_cs_n`     | 1            | Chip select; asserted on command issue               |
| `dfi_odt`      | 1            | DDR2 only; LPDDR2 held low                           |
| `dfi_ras_n`    | 1            | DDR2: RAS encoding; LPDDR2: held idle                |
| `dfi_reset_n`  | 1            | DDR2 only (DDR2 has no reset); LPDDR2 driven         |
| `dfi_we_n`     | 1            | DDR2: WE encoding; LPDDR2: held idle                 |

## Write Data Interface (MC → PHY)

| Signal              | Width                          | Notes                                  |
|---------------------|--------------------------------|----------------------------------------|
| `dfi_wrdata`        | `DFI_DATA_WIDTH`               | Write data; driven CWL cycles after WR |
| `dfi_wrdata_en`     | 1                              | High during valid write data beats     |
| `dfi_wrdata_mask`   | `DFI_DATA_WIDTH/8`             | 1 = mask byte; 0 = write (DFI convention) |

## Read Data Interface (PHY ↔ MC)

| Signal              | Width                   | Direction | Notes                                |
|---------------------|-------------------------|-----------|--------------------------------------|
| `dfi_rddata`        | `DFI_DATA_WIDTH`        | PHY → MC  | Read data, CL cycles after RD        |
| `dfi_rddata_valid`  | 1                       | PHY → MC  | High when beat is valid              |
| `dfi_rddata_en`     | 1                       | MC → PHY  | High when MC expects to capture read |
| `dfi_rddata_dnv`    | 1                       | PHY → MC  | LPDDR2 only: Data-Not-Valid          |

## Status Interface

Only `init_complete` is consumed:

| Signal              | Width  | Direction | Notes                                  |
|---------------------|--------|-----------|----------------------------------------|
| `dfi_init_complete` | 1      | PHY → MC  | High when PHY has finished its init    |

Other status signals (`dfi_data_byte_disable`, etc.) are not consumed.

## Sub-Interfaces NOT Driven

The following DFI v2.1 sub-interfaces exist in the port list but are tied to idle by `dfi_master`:

- **Update Interface** — `ctrlupd_req`, `phyupd_req`, etc.
- **Training Interface** — all rdlvl / wrlvl signals
- **Low Power Control Interface** — handled via CKE in command interface instead

## Reset

| Signal       | Width | Direction | Notes                                |
|--------------|-------|-----------|--------------------------------------|
| `dfi_rstn`   | 1     | MC → PHY  | DFI reset; controlled by init_engine |

## Timing Parameters Honored

The controller respects the following DFI v2.1 timing parameters:

- `tCTRL_DELAY` — PHY-reported command-launch delay (read from PHY's reported value; programmed via CSR if PHY has no advertise mechanism)
- `tPHY_WRDATA` — write data launch delay relative to WR command (default 0; programmable)
- `tPHY_RDLAT` — read data return latency relative to RD command (default = CL; programmable)

## Clock Domain

DFI signals are launched on `mc_clk` (the controller side). The PHY consumes them on `phy_clk` which runs at `N_PHASES × mc_clk` rate. The gear output stage handles the phase replication; the PHY handles the actual serialization to DRAM cycles.
